# 資料撰寫指南(M1)

## 我們在教模型做什麼

**任務**:輸入一段口語化 / 不夠正式的中文草稿,輸出潤飾成學術書面語的版本。
模型是 behavior cloning——它會直接模仿你給的 `polished` 版本的風格與品質上限,
所以每一筆資料的品質決定模型的天花板。

## Schema

最終每筆資料是一個 JSON object:

```json
{"category": "摘要潤飾", "draft": "口語草稿", "polished": "潤飾後版本", "source": "human_authored"}
```

- `category`:這筆屬於哪種學術寫作情境(見下方清單,也可自己加新類別)
- `draft`:潤飾**前**的樣子(口語、鬆散、有贅字、可能中英夾雜)
- `polished`:潤飾**後**的樣子(精簡、正式、符合期刊慣例)
- `source`:資料來源標籤(你親寫的一律 `human_authored`,轉換腳本自動補)

## 什麼是「好」的一筆資料

1. **draft 要像真的口語**:不要寫成「已經有點正式」的句子。draft 越接近你
   平常隨手寫、還沒潤飾的樣子,模型學到的對比訊號越強。
2. **polished 要是你真的會這樣寫**:這是關鍵——這批資料的價值就在於「真實
   的學術寫作經驗」。不要為了湊數寫你自己都覺得怪的句子。
3. **一次只示範一種轉換**:一筆資料最好聚焦一個潤飾重點(去贅字 / 轉正式
   語氣 / 補學術連接詞 / 專有名詞規範化),不要一句話同時改五件事,那樣
   模型學到的訊號會糊掉。
4. **繁體、台灣用語**:polished 一定要是繁體中文台灣慣用語(這是整個專案的
   差異化核心,也是我們踩過坑的地方——模型很容易滑向簡體)。
5. **長度別為了「正式」而膨脹**:我們之前發現 length bias(潤飾版常常比草稿
   長)。好的學術寫作經常是「更短更精準」,寫資料時有意識地示範「精簡」。

## category 參考清單(可擴充)

摘要潤飾、引言/研究動機、文獻回顧、方法段落、結果描述、圖表描述、
討論與限制、結論、研究貢獻、假設描述、操作型定義、研究問題陳述、
投稿信、Reviewer response、Email to advisor、致謝、研究計畫書用語、
白話轉學術、精簡有力、客觀語氣調整、專有名詞縮寫、統計結果報告、研究倫理

## 你的撰寫格式(低摩擦,不用手寫 JSON)

在 `data/raw/human_authored.txt` 裡,每筆用三行,類別用 `# ` 開頭:

```
# 摘要潤飾
draft: 我們這個實驗大概找了三十個人來測。
good: 本實驗共招募 30 名受試者。
```

- 每筆之間空一行(或不空都行,解析腳本會處理)
- `draft:` / `good:` 後面接內容(用 `good` 不用 `polished` 是因為打字快)
- 類別行 `# xxx` 之後的所有 pair 都屬於那個類別,直到下一個 `# yyy`

寫完後跑轉換腳本(見 M1 待辦:這支腳本你先寫、Claude review),
把 `.txt` 轉成帶 `source: human_authored` 的 `.jsonl`。

## 目標

**≥200 筆親寫**。可以分很多批,不用一次寫完。從你最熟的情境開始
(你自己論文的摘要、跟老師的 email、投稿信、reviewer response 都是
高價值素材——因為那是你真的寫過、改過的東西)。
